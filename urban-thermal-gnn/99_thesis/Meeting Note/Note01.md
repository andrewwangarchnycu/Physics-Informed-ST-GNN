## 問題 1：第二章內容過少（僅 8 頁）

### 一、 第二章內容過少的具體缺失

* **缺乏經典理論支撐：** 文獻回顧未從最經典的理論開始，例如 Kevin Lynch 的《城市意象》（Image of the City）等。
* **完全沒有文獻引用：** 此為最嚴重的缺失。寫法流於「說字化」（自說自話），缺乏引用。在論述過程中，每一小節起碼要引用 3 到 5 份研究以證明論點有所依據，否則讀者會質疑「憑什麼說這些話」。
* **論述流於「教科書」化：** 第二章不應只是介紹專有名詞、讀起來像教科書；而應透過分析他人的研究，指出其重點與問題，並說明哪些是本研究要採用的基礎。
* **缺乏地理脈絡與案例：** 目前內容完全未提及研究案例地點（新竹市）的地理背景，也未論述過去在新竹進行過的相關都市熱島研究。

### 二、 在章節結構與平衡中的角色

* **知識基礎的建立：** 第二章的功能是讓讀者理解本論文的建構基礎。雖然「整個抽掉不應影響論文核心研究的獨立性」，但它仍是理解後續研究不可或缺的「穿針引線」工作。
* **個人貢獻比例：** 整本論文中，學生自己的研究分量起碼要佔一半以上（包含第三、四、五章）。第二章主要放別人的東西，第三章則開始轉向自己的系統設計與分析。
* **圖表平衡：** 在涉及地圖與 GIS 的研究中僅放了 14 張圖，屬於嚴重不足。建議圖表應增加到 30 至 40 張以上，特別是系統圖、組織圖與資料架構圖，以減少過多冗長的文字描述。

### 三、 優化建議與調整方向

* **擴充頁數：** 應補充至十幾頁以上。
* **強化論證邏輯：** 透過引用文獻來「推卸責任」給前人研究，說明自己是站在巨人的肩膀上。
* **串聯研究問題：** 應針對第一章提出的研究問題，從理論、方法與脈絡（Context）的角度進行深入探討。
* **轉移技術細節：** 若某些技術細節或參數設定（如物理因子計算）過於繁瑣，應考慮移至第三章（系統設計）或附錄，以免干擾第二章的閱讀節奏。

> **小結：** 第二章的單薄反映了文獻深度不足與引用規範的缺失，學生需要重新建構理論框架，並透過增加經典理論、區域脈絡分析與圖表，使論文結構達到合理的平衡。

## 問題 2：標題與編號不統一

### 一、 標題與編號統一的規範要求

* **層級命名的一致性：** 對於第一章的命名，應決定是叫「序論」還是「第一章 序論」，並且後續的章節都要統一加註相同的標籤。
* **符號與格式的選取：** 對於第二章或其下小節的編號，使用「第二章」、「大寫二」或「2.」皆可，重點在於「統一就好」，避免讀者產生邏輯混淆。
* **避免過度細分編號：** 針對更細層級的小節（第三階小節），建議可以使用「小標籤（label）」的方式直接在段落開頭呈現，不需要強求寫出如「1.2.1」之類的四層編號，以維持版面的精簡與流暢。

### 二、 標題在章節結構中的邏輯角色

* **釐清研究範疇：** 質疑目前許多學生將第三章統一命名為「研究方法（Methodology）」的慣例。在論文結構中，第三章更應被定義為「系統分析與設計」，而「研究方法」應屬於第一章的子章節（如 1.2.3）。
* **區分他人貢獻與個人研究：** 統一且明確的章節結構能有效切分「別人的研究」與「自己的研究」。第二章應純粹放別人的文獻回顧，而從第三章開始（系統設計、分析）到第五章則全是學生自己的研究內容。
* **對應研究問題：** 標題設計應緊扣第一章提出的「研究問題」。透過第二章的文獻分析（對應理論、方法、案例）與第三、四章的系統實作與驗證，建立一個連貫的邏輯鏈條。

### 三、 結構平衡的優化策略

* **建立視覺框架：** 建議在第一章使用圖表（diagram/blueprint）來揭露整體的系統架構或研究問題關聯圖，讓讀者在進入細節章節前先有全觀概念。
* **減少冗長文字：** 透過增加圖表（如系統圖、組織圖、資料架構圖）至 30 至 40 張以上，來替代過多的文字論述，維持視覺與內容的平衡。
* **引用規範化：** 標題下方的論述必須有大量文獻支撐（每小節約 3 至 5 份引用），以避免「自說自話」的現象，確保論文結構建立在堅實的學術基礎上。

> **小結：** 標題與編號的統一是確保論文「可讀性」與「專業性」的第一步。透過標準化的層級命名與邏輯清晰的章節切分，才能讓讀者明確區分理論基礎（第二章）與實作貢獻（第三至五章），進而達成整本論文在質與量上的結構平衡。

## 問題 3：內容寫法太像教科書

### 一、 「像教科書」的具體定義與缺失

* **缺乏批判性分析：** 教科書提供的是公認的知識，但論文的第二章應該是研究者對他人研究的分析。需要指出前人研究的重點、存在的問題，並說明本研究計畫採用哪些部分作為基礎。
* **缺乏引用支撐：** 內容幾乎沒有引用，呈現出「說字化」（自說自話）的傾向。在論文中，必須透過引用來證明論點是有根據的，這也是「把責任推給別人」或「站在巨人肩膀上」的必要手段。
* **主題偏移：** 過多篇幅被用來解釋基礎科學細節（如歐幾里得幾何或基礎物理公式），這對於建築設計領域的研究而言過於瑣碎，且偏離了研究核心。

### 二、 在章節結構與平衡中的問題

* **喧賓奪主（比例失衡）：** 論文的核心應是作者自己的研究（通常占一半以上，即第三、四、五章）。如果第二章充斥著基礎教科書知識，會稀釋作者在後續章節中的原創論述。
* **章節功能錯置：** 第二章的角色應該是穿針引線，為理解後續研究建立必要的背景與基礎（Context）。若寫成教科書，則無法有效地將第一章的研究問題與第三章的系統設計串聯起來。
* **模糊研究邊界：** 當作者過度解釋通識技術時，讀者會搞不清楚這到底是工程研究還是設計研究。應將細節的技術操作或非核心的數據計算移至附錄，以維持本文的閱讀節奏與主軸平衡。

### 三、 優化與調整方向

* **強化分析視角：** 不要只是介紹專有名詞，應針對與研究密切相關的議題進行「分群（grouping）」，說明自己是基於誰的理論或方法進行改進。
* **納入地理與案例脈絡：** 第二章應包含特定區域（如新竹市）的地理脈絡分析及過去相關研究的評述，讓理論基礎與實際案例產生對話。
* **聚焦研究主軸：** 所有的文獻整理都應圍繞著「設計疊代」與「都市熱舒適」這兩個核心關鍵字。
* **適度的「科普」而非「授課」：** 針對跨領域讀者，僅需提供足以理解後續決策的必要資訊，不需追求讓每個人都懂 coding 或深奧的演算法細節。

> **小結：** 避免教科書式的寫法是為了確保論文的可解釋性與學術原創性。透過精簡通識內容並強化批判性對比，才能建立一個平衡且邏輯嚴密的章節架構。

## 問題 4：個人研究與他人文獻比例失衡

### 一、 第二章（他人文獻）的功能與缺失

* **缺乏學術支撐與自說自話：** 目前第二章僅有 8 頁「少的有點誇張」。缺失不僅在於頁數，更在於內容本質完全是「說字化」，缺乏引用。學術規範中需透過引用他人研究來證明論點有據，並將理論責任「推到別人身上」。
* **角色定位偏誤：** 第二章應定位為知識基礎的建構與「穿針引線」的工作，目的是讓讀者理解後續研究的理論背景（即使抽掉也不應影響核心研究的獨立性，它只是輔助理解的「脈絡化」過程）。目前將其寫成「介紹專有名詞」的教科書，缺乏對他人研究的分析、分群與評論，未能指出前人盲點。

### 二、 個人研究應佔比重（5:5 原則）

* **半數以上原則：** 整本論文中，作者自己的研究分量起碼要佔一半以上。
* **清晰的章節切分：** 論文結構應明確切分「別人的」與「自己的」。第二章是別人的文獻；而從第三章（系統分析與設計）、第四章（測試與驗證）到第五章，則應全是作者自己的開發與研究成果。
* **重新定義第三章：** 質疑將第三章統一命名為「研究方法（Methodology）」的寫法（這會讓內容流於程序描述），建議將其定位為「系統分析與系統設計」，從解決問題的決策過程出發，展現作者的思考深度。

### 三、 章節結構平衡的具體調整策略

* **強化「為什麼」的論述：** 論文不應只是交代做了什麼（How），更要解釋為什麼這麼做（Why）。作者應解構決策過程，說明為何選擇特定的演算法（如 PINN）或特定參數，這才是研究的精華。
* **圖表與文字的平衡：** 目前全論文僅有 14 張圖，對於地圖與 GIS 相關研究來說嚴重失衡。圖表數量應增加至 30 到 40 張以上（包括系統架構圖、資料架構圖、概念關聯圖等），以圖像化方式呈現複雜邏輯，減少冗長的文字。
* **技術細節的去留：** 過於瑣碎的技術細節（如基礎物理公式或特定的 coding 細節）若會干擾閱讀主軸，應考慮移至附錄，確保本文聚焦於「設計疊代」與「都市熱舒適」的主軸。

> **小結：** 健康的論文平衡應該是：以扎實的文獻回顧（第二章）為墊腳石，支撐起佔據全書一半以上篇幅、邏輯嚴密且圖表豐富的個人實作研究（第三至五章）。目前第二章過於單薄且缺乏引用，導致整體架構呈現「頭輕腳重」且原創論述不夠深入的問題。

## 問題 5：缺乏序論或章節導讀

### 一、 序論作為「研究藍圖」的功能

* **缺乏全局架構圖（Diagram/Blueprint）：** 第一章（序論）的核心功能在於提供讀者一個全觀的視角（Overview）與研究規劃的說明。第一章應該揭露整體的系統架構圖或研究藍圖，讓讀者在進入技術細節前，能先掌握研究問題與各章節間的關聯。
* **定位不明確：** 由於序論論述過於偏向「工程研究」，忽略了從「設計師的痛點」出發的動機說明，導致讀者無法理解研究的核心意圖（Intention）。
* **研究問題與假設的缺位：** 序論應包含明確的研究問題、前提與假設，目前版本的主軸不夠清楚。

### 二、 導讀在章節平衡中的必要性

* **建立「為什麼」的邏輯：** 序論扮演著「穿針引線」的角色。應先談設計師在面對都市熱島效應時的困難，再順勢提出為何需要透過技術（如 PINN）進行橋接。
* **圖表與文字的失衡：** 第一章目前完全沒有圖表，對於需要解釋複雜流程的研究來說是嚴重的結構性缺失。建議增加系統架構圖與關聯圖，以平衡過多的文字敘述並區分研究階層（區分第二章的前人基礎與第三至五章的原創貢獻）。

### 三、 章節結構與編號的統一性

* **標題與命名：** 第一章的命名必須與後續章節保持統一，例如統一使用「第一章 序論」而非混用不同的標籤。
* **明確的層級劃分：** 在序論中應妥善安排小節（如研究背景、動機、目標、範圍與限制），並透過統一的編號系統，確保論文結構的邏輯連貫性。

### 四、 優化與調整方向

* **新增系統架構藍圖：** 圖像化揭露研究問題與技術工序的關聯。
* **重整 1.1 節（研究動機）：** 從設計者的角度出發，說明「設計疊代」在現實中遇到的痛點，建立研究的必要性。
* **明確化研究範疇與限制：** 透過 1.4 節（範圍與限制）劃定研究邊界，作為論文的「停損點」，避免後續章節過度發散。

> **小結：** 序論不應只是開場白，而應是整本論文的「導航系統」。目前序論因缺乏圖表、動機論述不足及主軸不明，導致讀者難以理解後續章節的實作價值，進而破壞了整篇論文的結構平衡。

## 問題 6：第一章完全缺乏引註

### 一、 缺乏引註造成的「說字化」問題

* **缺乏公信力：** 第一章完全沒有引用，導致內容變成了作者的「說字化」（自說自話）。讀者會質疑作者：「憑什麼說這些話？」。在學術寫作中，任何論點都必須有所依據，不能僅憑個人直覺。
* **未能建構研究基礎：** 序論應說明研究是「建構在這些（前人的）基礎上」。缺乏引註會讓研究看起來像是憑空出現，而非學術脈絡的延續。

### 二、 引注在學術佐證中的「推卸責任」功能

* **責任轉移（Liability）：** 透過引用，作者可以將論點的理論責任「推到別人身上」，表明這是基於既有的研究成果，而非孤證。
* **累積前人研究：** 引用代表作者「累積別人研究，看很多書」，以此展現對該領域的通盤瞭解。

### 三、 文獻引用與佐證的具體要求

* **全方位引用：** 引用範疇應包含「別人的方法、別人的概念、別人的理論、別人的案例」。
* **技術性細節的佐證：** 即使是技術層面，也應引用「GitHub、白皮書、算法規範」等來源，證明技術選用是有根據的。
* **量化建議：** 每個小節起碼要有「三四份或四五份研究」作為佐證。雖然不需要「每一句話都引」，但核心論點與背景事實必須有扎實的文獻支撐。

### 四、 在章節結構中的負面影響

* **嚴謹度不一：** 學生之前的研討會論文（Conference paper）已經投過，理應更嚴謹，不應在學位論文中反而變得鬆散。
* **缺乏視覺支撐：** 第一章不僅缺引註，還「完全沒有圖表」，這使得序論在解釋複雜的研究藍圖時顯得單薄且缺乏說服力。

> **小結：** 第一章的引註缺失是不應該的。在文獻引用與佐證的脈絡下，作者必須透過大量引註來建立研究的權威性，將個人動機與學術界已有的知識體系掛鉤，從而避免淪為缺乏根據的自說自話。

## 問題 7：未將論點建立在他人研究基礎

### 一、 缺乏引用的「說字化」危機

* **缺乏公信力與混淆觀點：** 目前的初稿（尤其是第一、二章）幾乎沒有引用，導致論文陷入了「說字化」（自說自話）的困境。讀者會質疑作者「憑什麼說這些話？」，且若未明確加註引述，讀者會誤認為所有的論點都是作者自己的，這在學術規範中是不被允許的。
* **學位論文應有的嚴謹度：** 學生先前的研討會論文（Conference paper）已經發表過，學位論文的要求應該比那時更嚴謹，而非更鬆散。

### 二、 引用作為「學術責任」與「自保」的工具

* **推卸責任（Liability）：** 透過引用，作者可以將論點的理論責任「推到別人身上」，表示這是「別人說的」，自己只是負責累積並分析前人的研究。
* **站在巨人的肩膀上：** 引用的目的在於告訴讀者，這項研究是「建構在這些（前人的）基礎上」，顯示作者讀過很多書且對該領域有通盤了解。

### 三、 引用與佐證的範疇與要求

* **多元化的引用來源：** 必須加註「別人的方法、別人的概念、別人的理論、別人的案例」。
* **技術性佐證：** 即使是技術細節，也應引用 GitHub、白皮書或演算法規範等來源，證明技術選用並非憑空捏造。
* **量化要求：** 每個小節起碼要引用 3 到 5 份研究來支撐論述。

### 四、 文獻回顧（第二章）的功能重構

* **從介紹轉向分析：** 第二章（僅 8 頁）寫法太像「教科書」，正是未正確建立在他人研究基礎上的體現。論文不應只是「介紹」專有名詞，而應是「分析」他人的研究重點、發現其存在的問題，並明確指出哪些部分是本研究要採用的基礎。
* **穿針引線的定位：** 第二章的功能是為後續研究建立脈絡（Context），是讀者理解「研究基礎」不可或缺的環節。

> **小結：** 「引注」是論文的骨幹。作者必須透過廣泛引用經典理論與近代案例，將研究錨定在既有的知識體系中，以避免淪為缺乏證據的個人抒發，並確保整本論文的學術架構達到應有的平衡。

## 問題 8：技術來源未交代 (GitHub/白皮書)

### 一、 技術來源與引註的必要性

* **涵蓋多元來源：** 論文中的引用不應僅限於學術理論，技術實作的依據同樣必須明確交代。不論技術是從哪裡讀來的都必須引註，來源可以包括 GitHub 上的專案、組織釋出的白皮書（White Papers）、或是演算法的規範說明。
* **避免「說字化」：** 缺乏引用會讓內容流於自說自話，引發讀者質疑：「憑什麼說這些話？」。

### 二、 引註作為學術佐證的「責任轉移」功能

* **推卸責任（Liability）：** 透過引用 GitHub 或白皮書，作者可以將技術邏輯的正確性責任「推到別人身上」，表明這是基於既有的開發規範或他人驗證過的技術。
* **證明研究基礎：** 引用這些技術來源是為了告訴讀者，研究是「建構在這些基礎上」的，反映作者對該領域技術現況有充分的了解。

### 三、 在佐證體系中的定位

* **全方位佐證：** 技術引註是整體佐證體系的一環，必須與理論、案例併重，範疇應同時包含「別人的方法、別人的概念、別人的理論、別人的案例」。
* **具體化實作細節：** 即使是像 Web Socket 傳輸或是特定演算法（如 A* 或 GNN）的應用，也應交代決策背後的參考來源，而非僅視為理所當然的技術選擇。

### 四、 學術嚴謹度與誠信

* **釐清原創貢獻：** 這份學位論文應比之前的研討會論文（Conference Paper）更為嚴謹。若未交代技術來源（如 GitHub 上的現成代碼），讀者會誤以為這些都是作者的原創，這涉及學術誠信問題。作者應明確標註哪些是「參考別人的實作」，哪些才是自己的「原創貢獻（Originality）」。
* **量化引註要求：** 在涉及技術說明的章節小節中，起碼要有 3 到 5 份研究或技術文件作為佐證，以強化論述的公信力。

> **小結：** 技術來源的透明化是確保論文具備「可解釋性」與「可重現性」的基礎。透過對 GitHub 和白皮書的正確引註，作者不僅能證明自己的研究有所本，也能有效切分他人既有成果與個人開發貢獻，達成邏輯嚴密的章節平衡。
